# 额度重置手动触发

## 目的

提供定时触发之外的补救与提前发放入口：服务停机错过触发时点、或需要临时立即发放新预算时，管理员可立即执行一轮全员重置。

## 逻辑

管理端提供"立即按规则重置"操作：

1. 仅管理员角色可调用；普通用户无此入口，直接调用接口返回权限错误。
2. 执行范围与定时触发完全一致：遍历全体用户，对每个有生效规则的用户，按其生效规则的重置值执行重置；无生效规则的用户跳过。
3. 每个被重置用户的日志触发方式记为手动。
4. 手动触发与定时触发相互独立：手动执行不影响下一个定时触发时点，定时触发到点照常执行。重置为覆盖写，重复执行结果一致，不产生额外损益。

## 约束

- 非管理员调用手动触发接口必须被拒绝，且不发生任何用户余额变更。
- 手动触发对单个用户的余额结果与同一时刻定时触发的结果完全相同。
- 手动触发不修改任何规则配置，也不推迟、取消或提前任何定时触发时点。
- 手动触发一次，每个被重置用户产生恰好一条触发方式为手动的日志。

## 例子

周一 00:00 服务停机，weekly 全局规则（重置值 500000）该次未执行。周一 09:00 管理员在管理端点击"立即按规则重置"：

- alice（适用全局规则，当前 quota 20000）→ 重置为 500000，日志触发方式为手动。
- carol（带退出自动重置标记）→ 跳过，无日志。
- 下周一 00:00 定时触发照常执行，alice 再次被重置为 500000。

普通用户 bob 直接调用该接口 → 返回权限错误，所有用户余额不变。

## 验收

- [ ] 管理员触发后，所有有生效规则的用户被按各自规则值重置，无生效规则用户不变。
- [ ] 非管理员调用返回权限错误，且无任何余额变更与日志产生。
- [ ] 手动触发产生的日志触发方式为手动，逐用户一条。
- [ ] 手动触发后，下一个定时触发时点仍照常执行。
