# 保留字列名走统一引号常量

## 目的

避免每处 SQL 拼接各自硬编码保留字引号，让新增方言或漏改一处不再造成运行时 SQL 语法错误。

## 逻辑

关系型数据库对保留字列名（如 `group`、`key`、`user`）要求引号，且引号符号因方言而异：PostgreSQL 用双引号、MySQL/SQLite 用反引号。若每个查询片段各自硬编码，任一漏改都会在运行时报语法错误，静态难检出。

做法：库初始化时按当前方言一次性计算出「已加正确引号的保留字列名」字符串常量，供全仓所有拼 SQL 的地方引用。主库与日志库若可能是不同方言，各自提供独立常量。这些常量运行期不变。

## 约束

- 每个保留字列名有且仅有一个方言感知的常量作为拼接来源。
- 常量在库初始化时按当前方言判定填值，运行期不再变化。
- 任何 `Where` / SQL 拼接位置禁止直接书写 `"group"` / `` `group` `` / `"key"` / `` `key` `` 等硬编码引号形式。
- 主库与日志库若可能不同方言，各自的保留字常量互不混用。
- 规则可通过 grep 非初始化路径下的硬编码引号保留字命中数为 0 证伪。

## 例子

- 输入：多张表都有一个叫 `group` 的字段，需要在 `WHERE` 与 `SELECT` 中引用它。
- 过程：所有引用点写 `groupCol + " = ?"` 之类的拼接，`groupCol` 由初始化按方言填 `"group"` 或 `` `group` ``。
- 预期结果：同一份代码在三种方言下都能执行；新增第四种方言只需在初始化里加一个分支；无一处硬编码引号残留。

## 验收

- [ ] 存在方言感知的保留字列常量，且初始化路径唯一。
- [ ] 全仓 grep 非初始化路径下的 `"group"` / `` `group` `` / `"key"` / `` `key` `` 等硬编码形式，命中数为 0。
- [ ] 同一份代码在三种方言下都能通过冒烟 SQL 测试。
- [ ] 主库与日志库若可能不同方言，各自的常量不出现在对方库的 SQL 拼接中。
