# 方言判定走集中函数

## 目的

把「当前运行在哪种方言之上」这个判断收敛到唯一入口，让未来新增方言、拆分主库/日志库、或调整判定语义时无需扫遍全仓改 `==` 比较。

## 逻辑

同时支持多种关系型数据库的项目里，需要「按方言分叉」的代码会散落在 migration、SQL 片段拼装、内建函数选择等多处。若各处直接读全局方言变量再做 `==` 或 `switch` 比较，未来任何改动都是地毯式扫描。

做法：提供一个语义为「当前使用的是哪种方言」的判定函数（例如 `UsingMainDatabase(dialect)`），业务与迁移代码只调用它。方言常量的定义位置只出现在初始化与该判定函数中，其他地方不出现。

若主库与日志库允许各自独立配置方言，就分别提供两个判定入口（`UsingMainDatabase` / `UsingLogDatabase`），禁止混用。

## 约束

- 所有依方言分叉的运行时逻辑必须通过统一判定函数表达（例如 `UsingX(<dialect>)`）。
- 业务代码与迁移代码禁止直接读全局方言变量并与常量做 `==` / `switch` 比较。
- 主库与日志库若可能是不同方言，各自有独立判定入口，不得混用。
- 规则可通过 grep「方言常量名 `==`」在非判定函数文件下命中数为 0 证伪。

## 例子

- 输入：一段迁移逻辑需要在 PostgreSQL 下写 `ALTER COLUMN TYPE`、在 MySQL 下写 `MODIFY COLUMN`、SQLite 跳过。
- 过程：三个分支各自调用一次 `UsingMainDatabase(<该方言>)`，命中则拼对应 DDL。
- 预期结果：新增一种方言时，只需在初始化里加常量 + 新增一个分支；对已有方言行为无变化；不需要 grep 全仓改 `==`。

## 验收

- [ ] 存在一个语义为「当前使用某方言」的判定函数，全仓所有方言分支都通过它表达。
- [ ] 在业务与迁移代码里 grep 方言常量 + `==` 组合，命中数为 0（判定函数与初始化除外）。
- [ ] 主库与日志库若可能不同方言，各自的判定入口互不混用（grep 交叉命中数为 0）。
- [ ] 新增一种方言时，扩展点仅限于常量表 + 初始化 + 新分支，其他代码零改动。
