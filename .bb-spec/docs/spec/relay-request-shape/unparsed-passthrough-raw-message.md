# 未解析扩展字段用原始字节容器透传

## 目的

保证网关不消费语义的 provider 私有扩展字段在客户端与上游之间做到字节级别透传，不被「解码 → 内存中间态 → 再编码」的往返破坏。

## 逻辑

AI 网关聚合多家上游 provider，每家都有大量私有扩展参数（供应商专属开关、实验 flag、结构不稳定的对象）。网关无需理解其语义，只需保证「客户端传什么、上游收到什么」。

若把此类字段声明为具体 struct 或 `map[string]any`，会经过一次「解码 → 内存中间态 → 再编码」往返：数值精度可能变、字段顺序可能重排、未知子字段可能被 catch-all 抹掉；后续 provider 新增扩展字段还必须改 DTO 才能承接。

做法：把网关不解析语义的扩展字段声明为「原始 JSON 字节容器」类型（例如 `json.RawMessage`）配 `omitempty`——缺失时不出现、存在时原样透传。这条规则只覆盖「结构未知或非标量的扩展字段」；简单标量扩展（例如一个字符串 hint）保持标量类型即可。

## 约束

- 网关不解析语义的、结构未知或非标量的上游扩展字段必须以原始 JSON 字节形态存储（`json.RawMessage` 或等价类型）。
- 该类字段必须带 `omitempty`，缺失时不出现在上游请求 body 里。
- 禁止把此类字段声明为具体 struct 或 `map[string]any` 后又原样透传——引入无意义的解编码往返。
- 客户端发送的原始 JSON 值（数字精度、字段顺序、嵌套结构）必须在上游 body 中可等价识别。
- 新增一个网关不消费的 provider 扩展字段，唯一改动是在 DTO 加一行原始字节容器字段，无需实现任何转换逻辑。

## 例子

- 输入：客户端发送 `{ "provider_x_flag": {"nested": [1, 2, {"k": true}]}, "provider_y_hint": "beta-v3" }`，两字段网关并不消费。
- 过程：网关反序列化到请求 DTO，转发给上游 provider X 或 Y。
- 预期结果：上游收到的 JSON 里 `provider_x_flag` 的对象结构、嵌套数组、值类型、字段名与客户端输入等价；网关代码路径没有对这两个字段做任何解析或字段重命名。

## 验收

- [ ] 结构未知或非标量的 provider 扩展字段类型均为原始 JSON 字节容器。
- [ ] 该类字段 JSON tag 均带 `omitempty`。
- [ ] 数字精度、字段顺序、嵌套结构在客户端 → 上游全链路等价（可用差异测试断言）。
- [ ] 新增 provider 扩展字段时的 diff：DTO 仅新增一行字段，无转换代码。
